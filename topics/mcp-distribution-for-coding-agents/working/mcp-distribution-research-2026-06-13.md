# Ishoo MCP Distribution Research for AI Coding Agents

## Executive findings

The lowest-friction strategy for Ishoo is **not** “teach every user to edit MCP config,” and it is also **not** “make Ishoo the place where work starts.” The strongest pattern across the current host landscape is: install Ishoo once as a **per-user local executable**, then let each host consume that executable through its **native packaging surface** — plugin, marketplace entry, deeplink, extension, or user-scope config — so the user stays inside Claude Code, Cursor, Codex, or Google’s stack. The hosts with the cleanest current distribution surfaces for this are **Codex** and **Cursor**; **Claude Code** is very workable if you use plugins or skills-directory discovery instead of project `.mcp.json`; and **Google Antigravity** clearly supports one-click installation for supported MCP servers plus plugin-style packaging, but its public third-party distribution story is less explicit in the sources than Codex or Cursor. citeturn23view4turn26view1turn29view4turn30view0turn10view0turn16view1turn19view4turn3view6turn4view1turn46search2turn47search0turn48search0

For the “enable once, then forget it” experience you want, the most important design choice is to use **host-managed plugin/extension launch of a local stdio server** wherever possible. Cursor documents stdio as the local single-user mode it manages directly; Codex supports stdio servers in `config.toml` and in plugin-bundled `.mcp.json`; Claude Code plugin-bundled MCP servers start automatically when the plugin is enabled; and Gemini CLI / Google surfaces also support command-launched local MCP servers. That means Ishoo should ship a stable local `ishoo-mcp` binary and make every host-specific adapter point at that binary, instead of asking users to run a separate daemon manually or opening a long-lived local HTTP port unless a host specifically benefits from HTTP. citeturn10view0turn23view0turn3view6turn36view0

The other major conclusion is that **prompt suppression is only partially under your control**. Cursor asks for MCP tool approval by default unless tools are allowlisted or covered by Auto-review policy. Codex applies approval policies to MCP tools and separately requires trust review for plugin-bundled hooks. Claude Code project-scoped MCP/plugin flows hit workspace trust and, for project-scoped plugin MCP servers, per-server approval. Gemini CLI uses confirmation unless a server is explicitly marked `trust: true`, and if its Trusted Folders feature is enabled, untrusted workspaces will not connect to project MCP servers at all. In other words: Ishoo can remove configuration friction, but it cannot honestly remove every host’s security gates, and should not try to. citeturn10view0turn27view1turn26view3turn31view0turn5search1turn5search10turn36view0turn35view6

## Claude Code

**Verdict:** one-click-ish enablement is possible, but the best path for Ishoo is a **personal-scope plugin or skills-directory plugin**, not a checked-in project `.mcp.json`. Claude Code supports MCP servers configured directly, but it also supports **plugin-provided MCP servers** that can be declared in `.mcp.json` at the plugin root or inline in `plugin.json`. When a plugin is enabled, its MCP servers start automatically, their tools appear alongside other MCP tools, and they are managed through plugin installation rather than `/mcp` commands. Claude Code also supports **skills-directory plugins**: any folder under a skills directory that contains a `.claude-plugin/plugin.json` manifest is loaded on the next session with **no marketplace and no install step**, which is a powerful local-discovery mechanism for an installer-driven product. citeturn3view6turn4view1

Trust and persistence are where the product choice matters. Anthropic’s docs repeatedly distinguish **user scope** from **project/local scope**. Project or local MCP settings only run after the workspace trust dialog is accepted, and project-scope plugins inherit the same trust gate; for project-scope `@skills-dir` plugins, MCP servers also go through the same per-server approval as a project `.mcp.json`. By contrast, personal-scope plugins load from the user’s own Claude directories and avoid the project trust gate that governs repo-provided config. Anthropic also recommends central teams check `.mcp.json` into the codebase for shared integrations, but that is a **team rollout** pattern, not the lowest-friction individual install flow. citeturn5search1turn5search10turn4view1turn5search4

For Ishoo, the best Claude Code experience is therefore: install Ishoo, drop a **personal-scope Claude plugin** into the user’s Claude plugin/skills area, and let that plugin bundle the Ishoo MCP launch command plus any Claude-native skills or instructions you want. That gives you the “open Claude Code, see Ishoo there, enable once” feel without forcing a repo mutation or a workspace-trust detour. If you later need enterprise distribution, Claude’s docs also show plugin marketplace concepts in enterprise/GHES surfaces, but the strongest documented immediate path for a local-first commercial tool is still **installer-written personal plugin placement**. Also note one important limitation: local MCP availability matters only for local Claude Code sessions; Anthropic explicitly distinguishes cloud/web execution from sessions running on your machine, and says local MCP servers remain available in Remote Control because the session itself runs locally. citeturn4view1turn5search8turn5search3

## Cursor

**Verdict:** Cursor is one of the best hosts for Ishoo because it supports **three** strong enablement paths: official/team plugins, direct MCP install deeplinks, and ordinary `mcp.json` config. Cursor’s MCP docs say users can browse official plugins in the **Cursor Marketplace**, browse community MCP servers in **cursor.directory**, and click **“Add to Cursor”** on a marketplace entry to install it and authenticate with OAuth. Cursor also supports direct MCP registration through `.cursor/mcp.json` in a project or `~/.cursor/mcp.json` globally, and its docs expose an **extension API** for programmatic MCP server registration. On top of that, Cursor’s plugin docs expose a dedicated MCP deeplink format for install links, which is exactly the kind of “enable once” bridge Ishoo can use from its installer or website. citeturn10view0turn16view1

Persistence and approvals are also well documented. Cursor automatically uses MCP tools when relevant, but it **asks for approval before using MCP tools by default**. In Auto-review mode, allowlisted MCP tools can run immediately, and users or admins can preconfigure allowlists and autorun behavior in `permissions.json`. The editor and Cursor CLI share the same MCP configuration; the CLI can list, log in to, enable, and disable servers, and it automatically discovers configuration using project/global precedence. That means once Ishoo is installed in Cursor’s configuration surface, it stays available across Cursor’s local surfaces. citeturn10view0turn19view4

The biggest commercial caveat is Marketplace policy. Cursor’s plugin docs say marketplace plugins are distributed as Git repositories, are **manually reviewed**, and must be **open source**. Cursor also supports **team marketplaces**, and a plugin can be marked **required**, in which case it is installed automatically for everyone in a distribution group; Cursor’s pricing/enterprise materials also advertise **repository, model, and MCP access controls** on Enterprise. That makes Cursor excellent for enterprise rollout, but it also means an official public marketplace strategy may require Ishoo to publish at least a thin open-source adapter plugin even if the real value lives in Ishoo’s local binary. citeturn16view1turn21search6

For Ishoo, the practical Cursor recommendation is two-track. The immediate lowest-friction path is a **Cursor MCP deeplink** that registers the local Ishoo server without asking the user to edit JSON. The durable long-term path is a **Cursor plugin** that bundles MCP plus any rules/skills/hooks you need, because plugins are the only Cursor surface that packages instructions and behavior alongside the MCP server cleanly. For enterprise teams, the same plugin can later be distributed through a team marketplace and even marked required. citeturn16view1turn10view0

## Codex

**Verdict:** Codex is the single strongest host for Ishoo’s desired model. OpenAI has already converged on **plugins as the installable distribution unit**, and those plugins can include **skills, apps, MCP servers, and hooks**. Codex documents plugin browsing and installation in both the app and CLI, plugin detail/install deeplinks, local/repo/personal marketplaces, and shared MCP configuration across the app, CLI, and IDE extension. In plain terms: Codex already looks like the host most aligned with “install once, enable once, then the agent always has Ishoo.” citeturn28view0turn26view1turn23view0turn22search9

At the MCP layer, Codex stores MCP configuration in `~/.codex/config.toml` or project `.codex/config.toml` for trusted projects, and the CLI and IDE extension share that configuration. The Codex app settings page says users can enable **recommended servers** or add their own, and if a server requires OAuth the app starts the auth flow. Codex also lets users control MCP tool approval at the server or per-tool level with `default_tools_approval_mode` / `approval_mode`, using `auto`, `prompt`, or `approve`. That means Ishoo can either be installed as a raw MCP server, or — better — delivered as a plugin that bundles the MCP server and lets users or admins tune its approval posture afterward. citeturn23view0turn26view0turn27view1turn27view3

The plugin distribution surface is unusually rich. Codex supports a curated plugin directory, but it also supports **repo marketplaces** at `$REPO_ROOT/.agents/plugins/marketplace.json`, **personal marketplaces** at `~/.agents/plugins/marketplace.json`, **CLI-managed marketplaces** from GitHub/Git/local sources, and **deeplinks** like `codex://plugins/install/<plugin>?marketplace=<name>` or local plugin detail links with `marketplacePath=<absolute-marketplace-path>`. Codex installs marketplace plugins into a cache under `~/.codex/plugins/cache/...`, and stores their enabled/disabled state in `~/.codex/config.toml`. Crucially for Ishoo, OpenAI also documents that Codex can read a **legacy-compatible marketplace** at `$REPO_ROOT/.claude-plugin/marketplace.json`, and its plugin hooks set `CLAUDE_PLUGIN_ROOT`/`CLAUDE_PLUGIN_DATA` for compatibility. That creates a real opportunity to share parts of a Claude-style package layout with a Codex adapter. citeturn29view4turn30view0turn24search1turn24search11

The main guardrails are sensible and manageable. Codex only loads project-scoped config in **trusted projects**. Existing approval settings still apply when a plugin is installed. Plugin-bundled hooks are **not automatically trusted**; users must review and trust them. And on managed deployments, admins can use `requirements.toml` to enforce allowed approval policies, sandbox modes, MCP server allowlists, feature pins, and command rules across the app, CLI, and IDE extension. This is exactly the kind of enterprise control surface a commercial memory/plan/governance tool usually needs. citeturn23view1turn28view0turn23view5turn26view4turn31view0

For Ishoo, the cleanest Codex flow is: the installer writes a **personal marketplace** entry pointing at a local Ishoo plugin bundle, then opens a **Codex local plugin deeplink** so the user sees Ishoo in Codex’s normal plugin UX and enables it once. Because the same plugin concept spans app, CLI, and IDE extension, this gives Ishoo good coverage with one product integration instead of three separate ones. Public official publishing may become easier later, but OpenAI’s docs explicitly say self-serve official public plugin publishing is **coming soon**, so local/personal marketplace distribution is the best near-term route. citeturn29view4turn26view1turn30view0

## Google Antigravity and Gemini CLI

On Google’s current stack, I would treat **Antigravity** as the mainstream target and **Gemini CLI / Gemini Code Assist agent mode** as the compatibility path. Google’s Antigravity codelab describes Antigravity 2.0 as a standalone application for macOS, Linux, and Windows, alongside Antigravity IDE, Antigravity CLI, and the SDK. The same codelab shows onboarding that can optionally install plugins during setup. Antigravity’s MCP docs, as exposed in official search snippets, say users can browse supported servers, click **Install**, and then authenticate them from the **Customizations** tab. Antigravity’s settings docs split settings between **global** preferences and **project-level** boundaries, and its permissions docs let rules target exact MCP tools or all tools on a server. Its CLI/plugin docs also describe plugins as bundles of skills, agents, rules, MCP servers, and hooks, staged under a per-user home-directory path. Taken together, that is a solid host-native shape for Ishoo, even though the public third-party publishing mechanics are not as clearly spelled out in the parsed official docs as they are for Cursor or Codex. citeturn46search5turn46search2turn48search0turn47search1turn47search0

Gemini CLI remains important because Google says Gemini Code Assist’s VS Code **agent mode is powered by Gemini CLI** and exposes a subset of Gemini CLI functionality, including MCP. Gemini CLI supports local or remote MCP servers, user/project/system `settings.json` files, `gemini mcp add`, and `mcpServers` definitions that can include `command`, `url`, `httpUrl`, `includeTools`, and a server-level `trust` flag. If the Trusted Folders feature is enabled, the user gets a trust dialog the first time they open a folder, and untrusted workspaces will not connect to project MCP servers. Enterprise admins can centrally control settings and can enforce an allowlist with `mcp.allowed` plus system-level `mcpServers` definitions. citeturn35view5turn38view0turn40view3turn36view0turn35view6turn44view0

Gemini CLI also has a real extension surface. Extensions can be installed from a **GitHub URL or local path**, can be discovered through the extension gallery, and can bundle MCP servers directly in `gemini-extension.json`. Installed extensions are enabled globally by default unless disabled, and the extension gallery automatically indexes public repositories tagged appropriately. But Google’s docs also make two limits important for Ishoo: first, extension install is still fundamentally a **CLI command flow**, not a polished desktop marketplace flow; second, extension policies cannot use `allow` or `yolo` to auto-approve tool calls, so extensions cannot silently bypass security prompts. That keeps Gemini CLI viable as a fallback path, but it is not the best “consumer one-click plugin” story compared with Antigravity, Cursor, or Codex. citeturn40view0turn40view1turn40view2turn45view0turn45view2turn45view3

For Ishoo, the Google recommendation is therefore two-tiered. First, support **Antigravity** with a plugin/server-install experience that feels native to Antigravity’s Customizations / plugin surfaces. Second, keep a **Gemini CLI extension** available for users who work through Gemini Code Assist / VS Code or legacy Gemini CLI flows. If Google’s third-party Antigravity marketplace mechanics solidify, that mainline path should replace the CLI-first one; right now, based on the sources I reviewed, the Antigravity product clearly supports one-click installation for supported MCP servers, but the public documentation I found is less explicit about a third-party open marketplace/deeplink contract than the ones from Cursor or Codex. citeturn46search2turn46search5turn40view1turn40view2

## Recommended Ishoo product flow

The best cross-platform product flow for Linux and Windows is:

**Install a per-user Ishoo runtime, then stage host-native adapters, then launch the host’s own enablement UX.** In practice, that means the Ishoo installer should place a signed local `ishoo-mcp` executable in a stable **user-level** location, detect which agent hosts are installed, and then write only **user-scope adapter artifacts** for those hosts — not repo files, and not manual MCP snippets the user has to paste. That recommendation follows directly from the host capabilities above: user-scope/plugin flows avoid a lot of workspace-trust friction, while stdio launch is supported across the major hosts and keeps the host in charge of process lifecycle. citeturn10view0turn23view0turn3view6turn36view0turn4view1

For **Claude Code**, the adapter should be a **personal-scope plugin** or user skills-directory plugin that bundles Ishoo’s `.mcp.json` and any Claude-specific skills/instructions. For **Cursor**, the best immediate adapter is a direct **MCP install deeplink**, with a proper Cursor plugin as the long-term path if you want bundled rules/skills and enterprise distribution. For **Codex**, the best adapter is a **local plugin plus personal marketplace entry**, followed by a `codex://` deeplink into the plugin detail/install flow. For **Antigravity**, I would package a plugin/server install that lands in its normal plugin/customization surfaces, while keeping a **Gemini CLI extension** as fallback for users on Google’s CLI/Code Assist path. citeturn3view6turn4view1turn16view1turn10view0turn26view1turn29view4turn46search2turn47search0turn40view1

I would also strongly recommend a **two-layer instruction strategy**. Put host-agnostic guidance into the MCP server itself — especially server-wide instructions, prompts, resources, and tool metadata where the host consumes them — and put host-specific persistent behavior into plugins/extensions/rules/skills. Codex explicitly consumes MCP server `instructions`; Cursor supports tools, prompts, resources, roots, elicitation, and MCP Apps; Claude and Google both have parallel plugin/skill systems for persistent workflow guidance. For Ishoo’s concepts like project memory, scope contracts, ADRs, issue state, and completion gates, that dual-layer model will be more reliable than trying to squeeze everything into tool descriptions alone. citeturn23view0turn10view0turn16view1turn40view0

The unavoidable prompts should be surfaced as a **small, honest first-run checklist**, not hidden. Users should expect: Claude workspace trust if a project-scoped path is used; Cursor MCP tool confirmations unless allowlisted; Codex tool approvals per policy and hook trust review if hooks are bundled; Antigravity install/authenticate steps for supported servers; and Gemini CLI trust or confirmation flows depending on workspace trust and server trust settings. Ishoo should explain these as *host-native safety prompts* during enablement, then get out of the way. citeturn5search1turn5search10turn10view0turn26view0turn23view5turn46search2turn35view6turn36view0

If I turn this into a shipping priority list, it is:

First, build a **Codex plugin + local marketplace** path and a **Cursor deeplink/plugin** path. Those are the highest-confidence “normal app, enable once, then forget it” experiences today. Second, add a **Claude personal plugin** path that never relies on repo `.mcp.json` by default. Third, add **Antigravity** once your team is comfortable with Google’s newer plugin/customization surfaces. Fourth, keep **Gemini CLI / Code Assist** as a fallback extension-based path rather than the primary UX. That sequencing reflects both the richness of the documented packaging surfaces and the amount of friction each stack still imposes around trust, approval, or publishing. citeturn29view4turn26view1turn16view1turn10view0turn4view1turn46search2turn40view1

## Open questions and limitations

The biggest remaining uncertainty is **Google Antigravity’s public third-party publishing model**. The official sources I reviewed clearly show Antigravity 2.0, plugins, project/global settings, exact MCP permission targeting, and one-click installation for supported MCP servers, but they were less explicit than Cursor or Codex about a public third-party marketplace/deeplink contract for independent vendors. citeturn46search5turn46search2turn48search0turn47search1

There are also two ecosystem-specific limits worth keeping in mind. Cursor’s official marketplace requires plugins to be open source and manually reviewed, which may push Ishoo toward a thin open-source adapter plus proprietary local runtime. Codex’s plugin directory is already real, but OpenAI’s docs say self-serve official public plugin publishing and management are **coming soon**, so near-term go-to-market should assume **local/personal/repo marketplaces**, not a frictionless public app store. I also focused this report on the named hosts; I did **not** deeply profile OpenCode or KiloCode here. citeturn16view1turn30view0
